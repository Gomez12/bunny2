# i18n architecture

> Status: living document.
> Owners: phase-1.5 introduced this; later phases extend it.
> Source code: `apps/web/src/i18n/`, `scripts/i18n-check.ts`.

This document covers how user-facing strings are translated in bunny2,
why we picked the conventions we did, and how the discipline is
enforced automatically.

---

## 1. Goals

1. **No hardcoded user-facing strings.** Every visible label, button,
   placeholder, error, and aria-attribute is a translation key.
   (`AGENTS.md` §i18n is non-negotiable.)
2. **Server errors are localizable.** The server returns i18n keys, not
   English sentences, so the frontend renders them in the user's locale.
3. **Pipeline is testable.** A pre-commit / pre-PR check catches new
   hardcoded strings and missing keys before merge.
4. **Adding a locale is cheap.** Drop a JSON file, hook it into the
   bootstrap, done.

---

## 2. Stack

- [`i18next`](https://www.i18next.com) for the core key→string lookup.
- [`react-i18next`](https://react.i18next.com) for the `useTranslation`
  hook and the `t` function inside React.
- [`i18next-browser-languagedetector`](https://github.com/i18next/i18next-browser-languageDetector)
  for detecting the user locale from the browser (URL → cookie →
  navigator → fallback).

Bootstrap lives in `apps/web/src/i18n/index.ts` and is imported once
from `main.tsx` **before** the React tree mounts.

---

## 3. Namespaces & key conventions

Phase 1.5 uses a single translation bundle per locale with the
following top-level namespaces:

| Namespace | Used for                                                    |
| --------- | ----------------------------------------------------------- |
| `common`  | App-wide chrome: app name, loading, retry, tab labels.      |
| `status`  | Status page strings, including individual field labels.     |
| `chat`    | Chat page strings, including form labels and result labels. |
| `errors`  | Error keys, both client-side and server-emitted.            |

Keys are dot-separated and stable. Per `AGENTS.md`:

```text
good: chat.send, errors.chat.upstream, status.fields.dataDir
bad:  title, button1, errorMessage
```

When a server-side endpoint returns an error, the JSON body has an
`error` field whose value is an i18n key from the `errors.*` namespace
(e.g. `errors.chat.upstream`). The web client looks up that key via
`t(serverKey, { defaultValue: t('errors.network') })`, so an unknown
key degrades gracefully to a generic network error instead of leaking
an opaque code to the user.

---

## 4. Fallback rules

- `fallbackLng: 'en'` — English is the source of truth. Any locale
  missing a key falls back to the English string for that key.
- `supportedLngs: ['en', 'nl']` — phase 1.5 ships English (full) and
  Dutch (stub, to prove the pipeline).
- Missing keys in non-base locales are warnings from `i18n:check`, not
  errors. Missing keys in `en.json` are errors.

When phase 2+ adds production locales (e.g. `de`, `fr`), promote them
to "fully required" by tightening the check: `nl.json` will still be
allowed to be partial until it ships as a real product locale.

---

## 5. Discipline check (`bun run i18n:check`)

`scripts/i18n-check.ts` enforces three rules using the TypeScript
compiler API to parse `.tsx` (regex would false-positive). It exits
non-zero on any violation.

### 5.1 No hardcoded user-facing strings

Flags as errors:

- JSX text nodes that contain literal letters (after trim). Whitespace
  and pure-punctuation nodes pass.
- String-literal values for these user-facing JSX attributes:
  `placeholder`, `title`, `alt`, `label`, `aria-label`,
  `aria-description`, `aria-placeholder`, `aria-roledescription`,
  `aria-valuetext`.

Non-user-facing attributes (`className`, `href`, `id`, `role`, `type`,
`viewBox`, `xmlns`, …) are intentionally **not** flagged.

Allowed forms for visible content:

```tsx
<p>{t('status.title')}</p>
<Input placeholder={t('chat.placeholder')} />
<Button aria-label={t('common.tabs.chat')}>{t('common.tabs.chat')}</Button>
```

### 5.2 Used keys exist in `en.json`

Every static `t('…')` call is parsed; the literal key argument must
exist in `en.json`. Dynamic calls (`t(variable)`) are skipped — they
cannot be statically verified. To stay safe, frontend code that consumes
a server-returned key uses `t(key, { defaultValue: t('errors.network') })`.

### 5.3 `nl.json` integrity (warn-only)

- Keys present in `nl.json` but absent from `en.json` → warning. The
  Dutch locale is a stub, so this just catches typos.
- Keys present in `en.json` but absent from `nl.json` → warning. Phase
  1.5 intentionally ships partial Dutch.

---

## 6. Adding a new locale

1. Copy `en.json` to `apps/web/src/i18n/locales/<code>.json`.
2. Translate values; keep keys identical.
3. Import the file in `apps/web/src/i18n/index.ts` and add the code to
   `supportedLngs`.
4. Run `bun run i18n:check` — warnings only until the locale is
   declared "complete".
5. When the locale graduates to a shipped product locale, tighten the
   check (move it from warn to error in the script).

---

## 7. Accessibility tie-in

Translated strings are still strings — the accessibility of the
underlying component is independent. The patterns we enforce:

- Form inputs always have a paired `<Label htmlFor=…>` whose text
  comes from `t(...)`.
- Error states are announced via `role="alert"` on the element that
  renders `t(errorKey)`.
- The dev build wires `@axe-core/react` from `main.tsx` so a developer
  sees violations in the browser console during development; the
  import is dynamic and inside `if (import.meta.env.DEV)` so the prod
  bundle never includes it.

---

## 8. Chat keyboard convention

The chat textarea binds `Enter` to submit and `Shift+Enter` to insert a
newline. This is a common chat convention and is implemented in
`apps/web/src/pages/ChatPage.tsx::handleKeyDown`. Visible focus rings
come from the shared `focus-visible:ring-*` Tailwind utilities used by
every primitive in `apps/web/src/components/ui/`.

---

## 9. Future extensions (not in phase 1.5)

- Split the single bundle into per-screen namespaces once the catalog
  grows past ~200 keys (lazy load on demand).
- Pluralization: i18next supports it out of the box; introduce when a
  counted noun first appears.
- Server-side localization for emails / scheduled task notifications
  (phase 5+). Will reuse the same `errors.*` namespace conventions.
